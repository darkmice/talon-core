# Persistent Instruction: End-of-Task User Query (强制)

## 目的
确保所有面向用户的任务完成时，代理必须主动征询下一步动作，以保证流程清晰与可跟踪。

## 硬性规则（必遵）
- 代理在每次标记“任务完成”时**必须**使用 `AskQuestions`（或等价平台接口）向用户发起询问，且**不得**跳过或省略。无任何例外。
- 该询问**必须只调用一次**（per task completion），由调用方负责避免重复触发。
- 询问内容**必须**简洁、与上下文直接相关，并至少包含：
  - `multiSelect: true`（允许多选），
  - `allowFreeformInput: true`（允许自由输入），
  - 至少 3 个预设选项用于快速响应。

## 何为“任务完成”（用于判定触发时机）
代理须在下列任一清晰事件发生时触发 `AskQuestions`：
- 任务在任务清单或 TODO 中被标记为已完成；
- 与用户的交互产生确定性输出且用户确认“完成”；
- 代码/配置变更已提交并通过必要的预置校验（如 lint 或 CI）；
- 任务达到事先定义的里程碑并认为阶段性完成。

## 记录与存储（必遵）
- 用户在 `AskQuestions` 中的选择**必须**记录至会话内存路径 `/memories/session/`，用于后续上下文决策和审计；记录内容应包含时间戳、任务标识与所选项。
- 若需把选择写入长期存储或仓库（例如任务日志），必须先获得项目管理员授权并在团队文档中声明保存位置与保留期限。

## 错误处理与重试（必遵）
- 若 `AskQuestions` 调用失败（网络、接口异常等），代理**必须**按下列顺序处理：
  1. 立即重试最多 2 次（总共尝试 3 次），每次间隔短暂退避；
  2. 若仍失败，记录失败事件并把失败详情写入会话日志；
  3. 向用户发送一条明确通知，说明询问失败并提供三个默认备选项（例如："继续计划任务","复查输出","稍后再问"），同时再次尝试发起 `AskQuestions`；
  4. 将失败与后续用户响应一并记录到 `/memories/session/`。

## 模板与示例（强制格式要求）
必须使用如下模板或与之等价的 JSON 结构：

```json
{
  "questions": [
    {
      "header": "next-step",
      "question": "下一步要做什么？（可多选或自由输入）",
      "multiSelect": true,
      "allowFreeformInput": true,
      "options": [
        { "label": "制定任务计划", "description": "把工作拆成可执行任务" },
        { "label": "复查输出", "description": "检查变更或结果是否满意" },
        { "label": "运行测试", "description": "执行单元/集成/端到端测试" }
      ]
    }
  ]
}
```
 
## 约束与合规
- 本规则适用于所有面向用户的任务完成事件，代理实现必须遵守并以此为默认行为；团队可在项目级文档中说明如何对接或扩展。
- 代理不得以任何自动化、上下文或性能为由完全跳过 `AskQuestions` 的调用；在异常情况下应遵循“错误处理与重试”流程，确保最终向用户提供询问与记录。
